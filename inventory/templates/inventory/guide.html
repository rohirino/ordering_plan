画面説明・運用ガイド

販売実績、棚卸在庫、状態別資産、入荷予定をつなぎ、発注判断と棚卸報告を同じ商品コードで行うための実務ガイドです。

日次・月次の流れ画面一覧マスタ設定計算式と判定棚卸資産評価CSV取込運用チェック

1. 全体の運用フロー

1. 商品を整備商品コード7桁単位で、会社、LT、ロット、安全在庫、長期ベース、需要参照元を設定します。
2. 売上を更新日次販売履歴へ売上明細CSVを取り込み、需要予測の母数を更新します。
3. 入荷予定を更新到着予定CSVを1回取り込みます。商品コードから両社へ自動振分されます。
4. 月末棚卸を登録状態別・倉庫別に棚卸数量と原価を確定し、評価表を出力します。
5. 販売在庫だけを反映状態ごとの反映可否を確認して、棚卸数量を発注計画用在庫へ反映します。
6. 計画を確認120日先までの在庫推移、発注点割れ、欠品リスクを確認して発注計画を作成します。
会社の切替: 画面右上で日本育児とペットセレクトを切り替えると、それぞれの棚卸資産・在庫・入荷予定・発注計画を表示して管理できます。商品コードと商品名は共通の一覧として登録されるため、片方にすでにある商品でも販売履歴を取り込めます。棚卸CSVでは、状態コード 400、401、404 をペットセレクト分として扱います。

2. Dashboardごとの役割

画面主な役割通常の利用頻度判断時の注意
在庫・発注計画需要予測、在庫推移、発注点判定、発注計画の作成・進捗管理。日次ここに表示される在庫は、棚卸資産評価から反映した「販売在庫」です。
商品マスタ登録商品と発注パラメータ、状態コードマスタの登録・一括更新。随時LT、ロット、安全在庫、長期ベースは計算結果を直接変えます。
日次販売履歴売上明細の取込、売上数量・金額・粗利の確認。日次または週次需要予測は販売数量を使います。金額・粗利は確認用です。集計期間は全期間・直近7日・直近30日・当月・前月・指定期間で切り替えられます。
入荷予定・発注残確定・高確度・希望の入荷予定を管理。日次予測計算に加算されるのは「確定」「高確度」のみです。
棚卸資産評価状態別の数量・原価・在庫金額、倉庫別在庫、月次報告書。月次Ｂ品などを資産評価には含めつつ、発注計画用在庫からは除外できます。

一覧が複数ページになる画面では、最初・前へ・10前・指定ページ・10次・次へ・最後で移動できます。検索条件、会社、期間、並び順は維持されます。

3. 商品・状態マスタの設定

商品コードと状態コード

販売管理ソフトの10桁コードは、左7桁を商品コード、右3桁を状態コードとして扱います。商品マスタと販売需要は7桁単位、棚卸資産評価は「7桁商品コード + 3桁状態コード」単位です。

例: 0040008912 商品コード = 0040008 状態コード = 912 発注計画の需要・発注は商品コード 0040008 単位で判定
設定項目意味計算への影響設定の目安
所有会社資産・在庫・発注計画をどちらの会社へ計上するか。表示、棚卸、入荷予定、発注計画の所属を決めます。別会社の販売履歴を取り込んでも、この設定は自動変更されません。資産の帰属に合わせます。
需要参照元どちらの会社の販売履歴から需要を読むか。需要計算に使う売上実績の会社を決めます。販売実績を持つ会社を選びます。
LT発注してから販売可能在庫になるまでの日数。発注点 = 日次需要 × LT + 安全在庫。発注から倉庫入庫までの実績日数で設定します。
発注ロット発注可能な最小単位または丸め単位。必要数量をロット単位へ丸めます。仕入先の発注条件を設定します。
発注間隔分割発注する間隔（日数）。0 は従来どおり一度に発注します。正数を設定すると、間隔ごとに最小ロット以上で複数の発注計画を作ります。週次なら7日、隔週なら14日など、仕入先・発注業務の運用に合わせます。
安全在庫欠品を防ぐために残したい最低在庫。発注点に加算されます。需要のばらつき・供給不安定さが大きいほど増やします。
長期ベース90 / 120 / 150 / 180日の基礎需要期間。長期日次需要とABC/DEAD判定に使います。季節性や販売開始時期を考慮して選びます。
管理外発注判断の対象から外すフラグ。在庫・発注計画画面の計算と表示から除外されます。社内使用、サンプル、管理対象外に使います。
廃盤通常の一覧から隠すフラグ。原則として発注計画・一覧から非表示になります。終売後に設定し、必要なときだけ「廃盤も表示」で確認します。
状態コードの「発注計画反映」設定

状態別SKUごとに、棚卸数量を発注計画用の在庫へ含めるかを設定します。資産評価には全状態を残し、販売可能な状態だけを発注計画に流せます。

  • 含める: 販売でき、発注判断上も在庫として数えてよい状態。
  • 除外: Ｂ品、破損品、保留品、展示品など、資産としては残すが通常販売在庫に数えない状態。
  • 「表示中を含める／除外」で、状態コード・状態名のフィルタ結果だけを一括変更できます。

4. 発注計画の計算式と判定根拠

計画の基準日は、対象会社の今日以前の最新販売実績日の翌日です。販売実績がない場合は需要予測の対象外とし、通常表示から除外します。この基準日から120日間を日ごとに試算します。

4-1. 基礎需要とトレンド倍率

商品ごとに設定された長期ベース日数を T（90 / 120 / 150 / 180日）、その期間の通常販売数量を S_long、直近30日の通常販売数量を S_30 とします。日次販売履歴で「先付け」に指定した明細は、ここでいう通常販売数量には含めません。

長期日次需要 D_long = S_long / T 直近日次需要 D_30 = S_30 / 30 生トレンド R = D_30 / D_long ただし D_long = 0 のときは R = 1.00 適用トレンド M = min(トレンド上限, max(トレンド下限, R)) 予測日次需要 D = D_long × M 初期値: トレンド下限 0.50、トレンド上限 2.00

つまり、直近の販売が長期平均より速ければ予測需要を増やし、鈍ければ減らします。ただし極端な一時変動で予測が振れすぎないよう、商品ごとの上限・下限で止めます。

画面の「中期」は長期ベースの半分です（90日なら45日、120日なら60日、150日なら75日、180日なら90日）。長期・中期・30日の各値は、比較しやすいよう月換算で表示します。中期は傾向確認用であり、発注計算そのものは長期ベースと直近30日を使用します。

例: 90日で900個、直近30日で450個なら、D_long = 10個/日、D_30 = 15個/日、R = 1.50です。上限2.00・下限0.50なら、予測日次需要は 10 × 1.50 = 15個/日 になります。
先付け受注の考え方: 固定の数量・金額で「大口」と自動判定はしません。通常の販売ペースとは別に、特定の日に在庫を引き当てる必要があるイレギュラーな大口受注を、担当者判断で先付け受注として登録します。大口に見えても定期的に発生する受注は、需要トレンドへ反映させるため先付けには登録しません。

4-2. 発注点

発注点 OP = 予測日次需要 D × リードタイム LT + 安全在庫 SS

発注してから入荷するまでに消費する見込み数量と、入荷後も残しておきたい安全在庫の合計です。現在庫または将来予測在庫がこの水準以下になると、発注点割れとして表示します。

例: 予測日次需要15個、LT30日、安全在庫100個の場合、発注点は 15 × 30 + 100 = 550個 です。

4-3. 120日先までの在庫推移

翌日の予測在庫 = 前日の予測在庫 + 入荷予定数量（確定 + 高確度） - max(出荷予定数量, 予測日次需要 D)
  • 入荷予定: 「確定」と「高確度」を加算します。「希望」は画面に表示しますが、計算には加算しません。
  • 先付け受注: 在庫・発注計画の「勢い（トレンド倍率）状況」の右にある「先付け受注」で、商品ごとに出荷日・向け先・数量を登録します。登録数量は、その日の出荷予定数量として在庫予測へ反映されます。
  • 消費数量: 出荷予定が登録されている日は、出荷予定と予測日次需要の大きい方を消費として採用します。
  • 欠品リスク: 120日間のどこかで予測在庫が0以下になった場合。
  • 発注点割れ: 欠品には至らないが、予測在庫が発注点以下になった場合。
  • 期限表示: 初めて0以下になる予測日からLTを差し引き、基準日から見て何日以内に発注判断が必要かを表します。
重要: この画面は発注判断を支援する予測です。先付け受注に未登録の大口受注、返品、倉庫間移動、生産遅延は自動では反映されません。判断前に入荷予定と特記事項を確認してください。

4-4. 先付け受注と販売履歴の扱い

  1. 将来の特定日に出荷するイレギュラーな大口受注は、在庫・発注計画の商品行で「先付け受注を登録」を開き、日付・向け先・数量を登録します。定期的な大口受注は登録せず、通常の需要トレンドで管理します。
  2. 同じ受注が販売履歴CSVにも含まれている場合は、日次販売履歴Dashboardで該当明細の「先付け」にチェックを入れて保存します。
  3. 先付けにした販売履歴は、長期・中期・直近30日の販売実績、トレンド倍率、ABC/DEAD判定、発注計算から除外されます。数量・金額の履歴表示には残ります。
  4. 受注が取り消し・変更になった場合は、在庫・発注計画の先付け受注明細を削除または登録し直し、必要に応じて販売履歴の先付けチェックも戻します。

4-5. 自動作成する発注数量

120日予測の最小在庫を MinStock としたとき、発注点まで戻す不足数を計算します。

不足数 Shortage = max(0, OP - MinStock) ロット積上げ: 発注数 = ceil(Shortage / Lot) × Lot 最小ロットのみ: 発注数 = max(Lot, ceil(Shortage))

「ロット積上げ」はロット倍数に切り上げます。「最小ロットのみ」は不足数がロットを超える場合に不足数そのものを発注し、ロット未満だけを最低ロットまで引き上げます。作成された計画はまず「計画中」で登録され、確認後に「発注済」、入庫後に「入庫済」へ更新します。

分割発注: 商品マスタの発注間隔が0日以外の場合、発注計画は設定間隔ごとに在庫見込みを確認し、LT後の入荷予定日まで在庫が不足しそうなときに最小ロット以上の計画を追加します。各計画には発注予定日と入荷予定日が表示され、入荷予定日には在庫タイムラインに 計:数量 と表示されます。

4-6. 安全在庫の提案値

画面に表示される提案値は自動保存されません。設定見直しの参考値です。

ばらつき V = max(|D_30 - D_long|, D × 0.30) 提案安全在庫 = ceil(1.65 × V × sqrt(LT)) 表示範囲: 最低10個、最高300個

1.65は、おおむね95%水準のサービスレベルを意識した係数です。季節商品・輸入遅延・欠品許容度など、数式に入らない事情は担当者判断で調整してください。

4-7. ABCランクと処分推奨

  • 廃盤でない商品を、各商品の長期ベース期間の販売数量が多い順に並べます。
  • 累積販売構成比が70%以下をA、95%以下をB、それ以外をCとします。
  • 設定した長期ベース期間の販売が0で、現在庫が1個以上ある商品は DEAD（処分推奨） に上書きします。
  • DEAD商品は通常発注を止めます。「例外発注を許可」を明示的にONにしたときだけ、発注候補・KPIに含めます。

5. 棚卸資産評価と発注計画在庫の連携

棚卸資産評価の計算

状態別・倉庫別在庫金額 = 棚卸数量 × 状態別原価 商品別在庫金額 = 状態別・倉庫別在庫金額の合計 倉庫別在庫金額 = その倉庫に属する明細の合計

同じ商品でも状態コードが異なれば、原価・数量・在庫金額を別々に保持します。棚卸日ごとの原価は履歴として残るため、月ごとの評価額推移を追えます。

棚卸CSVの雛形では、状態別評価原価に、状態付き10桁コードごとの評価原価を円単位で入力します。元表の移動平均原価に小数がある場合は、取込時に四捨五入して円単位にします。

棚卸CSVから発注計画へ反映する手順

  1. 対象会社と棚卸日を選び、棚卸CSVを取り込みます。
  2. 未登録商品は商品マスタに自動生成されます。商品名がCSVと異なる場合は更新履歴も残ります。
  3. 状態マスタCSVを取り込み、状態名を整えます。
  4. 状態別明細をフィルタし、販売に数えない状態を「除外」にします。
  5. 「発注計画へ数量反映」を実行します。
  6. 在庫・発注計画で現在庫と倉庫別内訳、欠品リスクを確認します。
発注計画へ反映する商品在庫 = 同じ商品コードのうち「発注計画反映 = 含める」の状態別・倉庫別数量合計
反映ボタンは、選択した棚卸日・会社の在庫を発注計画用在庫として更新します。棚卸日や会社を取り違えないよう、実行前に上部の選択状態を確認してください。

使わなくなった倉庫は「倉庫別在庫金額」の在庫ゼロの倉庫を整理から一覧と棚卸CSV雛形から外せます。過去の棚卸履歴は削除されず、後の棚卸CSVに同じ倉庫名の列を含めれば再び一覧へ戻ります。

7. 日次・月次の実務チェックリスト

日次または週次

  1. 日次販売履歴に最新の売上明細CSVを取り込み、取込ログで警告件数を確認します。警告があれば「スキップCSV」で日付・得意先コード・商品コードなどのスキップ理由を確認します。
  2. 先付けとして扱う受注があれば、在庫・発注計画へ日付・向け先・数量を登録し、販売履歴側の該当明細も「先付け」にします。
  3. 入荷予定・発注残に最新の到着予定表を取り込み、未登録商品・日付不正がないか確認します。
  4. 在庫・発注計画で、欠品リスク、発注点割れ、DEAD商品を確認します。
  5. 必要な商品だけ「発注」を実行し、計画中一覧を確認後に発注済へ進めます。
  6. 分割発注の商品は、発注予定日・入荷予定日・数量が複数行で作られていることを確認します。間隔や最小ロットを変える場合は、商品マスタまたは商品行の設定から変更して再計算します。
  7. 実際に入庫したら、該当の計画を入庫済へ更新します。

月次（棚卸日）

  1. 棚卸CSVと状態マスタCSVを準備します。
  2. 棚卸資産評価へ取り込み、在庫数量合計・在庫金額合計・倉庫別在庫金額を確認します。
  3. Ｂ品・保留品などの「発注計画反映」を確認します。
  4. 発注計画へ数量反映を実行します。
  5. 在庫・発注計画で基準日、現在庫、120日予測を確認します。
  6. 必要に応じて棚卸資産評価のExcel/PDFを報告書として出力します。

分割発注の計画が想定どおりでない商品マスタの発注間隔（0は一括）、発注ロット、LT、既存の計画中・発注済の入荷予定日を確認します。検証シミュレーターでも同じ間隔・ロットを入力して在庫線を確認します。
運用の原則: 商品・販売・入荷・棚卸の各データは専用Dashboardで更新し、Django管理画面は緊急修正や保守確認に限定します。変更前後の取込ログと出力帳票を残すと、数値差異の追跡が容易になります。
